# Go Codex App Server Client Design

## Goal

Add a small, testable Go Codex app-server client that can launch a local
`codex app-server` subprocess, initialize the protocol, start one thread, run
one turn, and emit enough structured events for a later orchestrator integration.

## Scope

This first implementation does not replace `agent.command` in the Go
orchestrator. It builds the app-server client behind a package boundary so the
protocol can be verified with fake app-server processes before it affects
workspace execution.

In scope:

- Local stdio JSON-RPC client for Codex app-server.
- Protocol startup using `initialize`, `initialized`, `thread/start`, and
  `turn/start`.
- One active turn per client session.
- Structured events for session start, turn completion, notifications, token
  usage, malformed protocol lines, server requests, and process exit.
- Fake-process tests that validate bytes sent to and from the client.

Out of scope for this first change:

- Replacing `agent.command` in the orchestrator.
- Running real Codex in CI.
- SSH app-server launch.
- Dynamic tool execution such as `linear_graphql`.
- Approval auto-resolution beyond classifying server requests as needing input.
- Making app-server mode the default for `agent.kind: codex`.

## Protocol Source

The implementation targets the local Codex CLI schema generated by:

```bash
codex app-server generate-json-schema --experimental --out <tmpdir>
```

Local version checked during design: `codex-cli 0.133.0`.

The schema confirms these core client requests:

- `initialize`
- `thread/start`
- `turn/start`
- `turn/interrupt`

The first client only needs the first three. The schema also confirms the
server notifications needed for dashboard and health integration later:

- `thread/started`
- `turn/started`
- `turn/completed`
- `thread/tokenUsage/updated`
- `item/agentMessage/delta`
- `item/reasoning/textDelta`
- `item/plan/delta`
- `error`

Server requests such as `item/commandExecution/requestApproval`,
`item/fileChange/requestApproval`, `item/tool/requestUserInput`, and
`item/tool/call` are treated as input-required events in this phase.

## Architecture

Create `go/internal/codexappserver` with three responsibilities:

- `Client`: public orchestration API for `Start`, `RunTurn`, and `Close`.
- `process`: subprocess launch and shutdown for `bash -lc <command>`.
- `rpc`: newline-delimited JSON-RPC request/response handling over stdio.

The package returns events through a callback:

```go
type Event struct {
    Type      EventType
    Time      time.Time
    ThreadID  string
    TurnID    string
    Method    string
    Message   string
    Usage     *TokenUsage
    Raw       string
}
```

The client owns protocol details and does not import orchestrator packages. The
orchestrator integration can later translate these events into existing running
entry fields: session id, app-server pid, last event, last message, token totals,
and run-health progress.

## Data Flow

1. Caller creates a client with command, workspace, timeout, and event callback.
2. Client launches `bash -lc <command>` in the workspace.
3. Client sends `initialize`.
4. Client waits for the matching JSON-RPC response.
5. Client sends `initialized`.
6. Client sends `thread/start` with cwd, sandbox, approval policy, and optional
   runtime workspace roots.
7. Client stores the returned `thread.id`.
8. Client sends `turn/start` with the prompt as text input.
9. Client reads newline-delimited protocol messages until `turn/completed`,
   process exit, timeout, malformed critical response, or input-required request.

## Error Handling

- A JSON-RPC error response for `initialize`, `thread/start`, or `turn/start`
  fails startup with method context.
- A malformed line while waiting for a specific response fails startup.
- A malformed line during streaming emits `EventMalformed` and continues.
- Server requests that require approval, tool execution, or user input return
  `ErrInputRequired` and emit an input-required event.
- Process exit before turn completion fails the turn.
- Context cancellation kills the subprocess and returns the context error.

## Testing

Use only fake app-server processes in unit tests. Tests should assert both
outgoing client messages and incoming event handling.

Required tests:

- initialize and thread start send valid JSON-RPC requests in order.
- `RunTurn` sends a text input turn and returns on `turn/completed`.
- token usage notifications populate `Event.Usage`.
- server approval/tool/user-input requests return `ErrInputRequired`.
- malformed streaming lines emit `EventMalformed` but do not abort the turn.
- process exit before completion returns an error.

## Rollout

This should land as a narrow PR that adds the package, tests, and README/spec
notes only. A follow-up PR should introduce a runner abstraction in the Go
orchestrator and wire this client behind an opt-in config.
